## Prompt

> I'm not sure how I would implement different operation types. They can more or less be structured in e.g. Drilling, milling, turning, ... But every type has different subtypes (eg pocketting milling or spigot milling, drilling/pecking/reaming/tapping). It's getting rather complex to keep the subdivision in categories as 'Drilling' as they will have different parameter signatures. How can we tackle this problem? Propose some methods to refactor the different operation types.

## Current State (as of 2026-03-28)

The current code uses a two-part structure:

**`src/core/operation.rs`** defines:
- `OperationType` — a simple enum: `Drilling`, `Milling`
- `Operation` — a single struct holding all fields generically (`name`, `operation_type`, `tool`, `global_clearance`, `capabilities`, `locations`)
- `DrillStrategy` — a sub-type enum: `Manual`, `General`
- `OperationLocations` — an untagged enum: `Points { ... }` or `Pattern { ... }`

**`src/io/job.rs`** defines:
- `OperationConfig` — a `#[serde(tag = "type")]` enum with `Drilling { common, strategy, locations }` and `Milling { common, strategy }`
- `CommonOperationConfig` — shared YAML fields (`name`, `tool_number`, `tool_name`, `tool_diameter`, `spindle_speed`)

**`src/nc/compiler.rs`** contains `process_drilling()` which matches on `DrillStrategy` and dispatches to `compile_manual_drill()`.

**`src/io/job.rs` `run_job()`** iterates over operations, matches on `OperationConfig`, constructs an `Operation` struct from the common fields, and calls the compiler.

### The problem

As new operation types are added (peck drilling, reaming, tapping, pocket milling, profile milling, facing, turning, ...), the parameter signatures diverge significantly:

| Operation | Specific parameters |
|-----------|-------------------|
| Drill | strategy, locations, depth |
| Peck | locations, depth, peck_depth, retract_mode |
| Ream | locations, depth, feed_out |
| Tap | locations, depth, pitch |
| Pocket | geometry, stepover, stepdown |
| Profile | geometry, offset_side, stepdown |
| Face | geometry, stepover |

The current approach of a broad `OperationType` category enum + a `strategy` sub-field becomes awkward: "drilling" contains operations (ream, tap) that share locations but differ in every other parameter. Grouping by category adds a layer that doesn't help the compiler or the data model.

## Decision: Shared struct + kind enum + trait for behavior

Common fields live in a single `OperationCommon` struct. Type-specific parameters live in per-type structs held by an `OperationKind` enum. Behavior is defined by an `OperationType` trait that each per-type struct implements. The enum preserves exhaustive matching for serde and wiring; the trait keeps behavior encapsulated per operation type.

### Data model

```rust
pub struct Operation {
    pub common: OperationCommon,
    pub kind: OperationKind,
}

pub struct OperationCommon {
    pub name: String,
    pub tool: Tool,
    pub clearance: f64,
    pub skip_level: Option<u8>,
}

pub enum OperationKind {
    // Hole-making
    Drill(Drill),
    Peck(Peck),
    Ream(Ream),
    Tap(Tap),

    // Milling
    Pocket(Pocket),
    Profile(Profile),
    Face(Face),
}

// Per-type structs hold only type-specific fields
pub struct Drill {
    pub strategy: DrillStrategy,
    pub locations: Locations,
    pub depth: f64,
}

pub struct Peck {
    pub locations: Locations,
    pub depth: f64,
    pub peck_depth: f64,
    pub retract: RetractMode,
}

// ... etc
```

### Behavior via trait

The `OperationType` trait defines the interface every operation type must implement:

```rust
pub trait OperationType {
    fn generate(&self, common: &OperationCommon) -> Vec<ToolpathSegment>;
    fn compile(&self, common: &OperationCommon, segments: &[ToolpathSegment]) -> Vec<NCBlock>;
    fn validate(&self, common: &OperationCommon) -> Result<(), Error>;
}
```

Each per-type struct (`Drill`, `Peck`, `Pocket`, ...) implements `OperationType`. The struct owns its specific parameters and the trait methods receive `&OperationCommon` for shared fields.

### Dispatch via `kind_impl`

A single private method on `Operation` bridges the enum to the trait. This is the only place the enum match exists:

```rust
impl Operation {
    fn kind_impl(&self) -> &dyn OperationType {
        match &self.kind {
            OperationKind::Drill(d) => d,
            OperationKind::Peck(p) => p,
            OperationKind::Pocket(p) => p,
            // ... compiler forces a match arm for every variant
        }
    }
}
```

Public methods delegate through `kind_impl` — callers never see the trait or the enum:

```rust
impl Operation {
    pub fn generate(&self) -> Vec<ToolpathSegment> {
        self.kind_impl().generate(&self.common)
    }

    pub fn compile(&self, segments: &[ToolpathSegment]) -> Vec<NCBlock> {
        self.kind_impl().compile(&self.common, segments)
    }

    pub fn validate(&self) -> Result<(), Error> {
        self.kind_impl().validate(&self.common)
    }
}
```

Adding a new operation type means:

1. Define a struct with its specific fields.
2. Implement `OperationType` for it — `generate()`, `compile()`, `validate()`.
3. Add the variant to `OperationKind` — the compiler forces a match arm in `kind_impl`.

The match in `kind_impl` is mechanical (one line per variant). All real logic lives in the trait implementation, encapsulated per type.

### Why this design

- **Common fields in one place.** `operation.common.tool` works without matching. No repetitive destructuring.
- **Exhaustive matching preserved.** The enum is still the backbone for serde and wiring. Adding a variant forces a compiler error in `kind_impl` — guaranteed.
- **Encapsulated behavior.** Each operation type is self-contained: its struct holds specific data, its trait impl holds specific logic. No growing match arms across multiple methods.
- **Single dispatch point.** The enum match exists once in `kind_impl`, not duplicated across `generate()`, `compile()`, `validate()`, etc.
- **Scales with operation count.** 20 operation types means 20 one-line arms in `kind_impl` — not 20 arms × N methods.
- **YAML ergonomics work.** `#[serde(tag = "type")]` on `OperationKind` + `#[serde(flatten)]` for common fields maps directly to the current YAML format.
- **Category grouping is cosmetic.** "Drilling vs milling" can be a derived method on `OperationKind` or a comment in the enum — not a structural layer.

### Tradeoffs accepted

- Type-level enforcement of per-operation constraints (e.g. "tapping requires CW spindle") is not possible — these become runtime validation in `validate()`.
- All shared fields in `OperationCommon` are accessible on every operation, even if not all are equally meaningful for every kind. In practice, tool + clearance + name are universal.
- One level of indirection via `&dyn OperationType` in `kind_impl`. This is a single vtable lookup — negligible cost for a CAM tool where I/O dominates.

## Analysis

### Which problem are we actually solving?

The core tension is: operations share some fields (tool, clearance, name, skip level) but diverge in their specific parameters. The question is where to draw the line between shared and specific.

### Category grouping is cosmetic, not structural

The original instinct to group by category (Drilling, Milling, Turning) is natural from a machinist's perspective but doesn't help the data model. A "ream" and a "drill" share locations but differ in every other parameter. A "pocket" and a "profile" share geometry but differ in strategy. The category is metadata, not structure.

If categories are useful for display (CLI output, future UI grouping), they can be derived from the variant or stored as a simple tag — they don't need to be a layer in the type hierarchy.

### Exhaustive matching matters here

The operation set is closed at compile time. When a new operation type is added, the compiler should force updates to: (1) the YAML deserializer, (2) the dispatch wiring. Pure trait objects give up this guarantee — but the hybrid approach preserves it. The enum in `OperationKind` forces a compiler error in `kind_impl` when a new variant is added, while the trait ensures the new type implements `generate()`, `compile()`, and `validate()` before it compiles.

### YAML ergonomics constrain the design

The YAML job file is the primary user interface. Whatever internal model is chosen, it needs to deserialize cleanly from:

```yaml
operations:
  - type: drill
    strategy: manual
    tool_number: 1
    spindle_speed: 1200
    points:
      - [25.0, 15.0]

  - type: pocket
    tool_number: 3
    stepover: 0.6
    stepdown: 2.0
    geometry: outline.svg
```

Serde's `#[serde(tag = "type")]` with internally tagged enums handles this naturally.

## Refactoring Plan

> **Disclaimer:** This plan was generated by Claude (Sonnet 4.6 / Opus 4.6), an AI assistant by Anthropic. It reflects the agreed-upon design from the discussion above and the current code state as of 2026-03-28. Intended as a guide for the project author to implement.

### Context

Operations are currently spread across three modules: type definitions in `core/operation.rs`, toolpath generation in `toolpath/drill.rs`, and NC compilation in `nc/compiler.rs`. This means every new operation type touches three places. The design above consolidates this into a single `operations/` module with a shared `Operation` struct, an `OperationKind` enum, and `generate()`/`compile()` methods on `Operation` that dispatch to per-type modules.

### Step 1 — Create `src/operations/mod.rs`

Define the shared types, trait, and dispatch:

```rust
pub struct OperationCommon {
    pub name: String,
    pub tool: Tool,
    pub clearance: f64,
    pub capabilities: PostprocessorCapabilities,
}

pub trait OperationType {
    fn generate(&self, common: &OperationCommon) -> Vec<ToolpathSegment>;
    fn compile(&self, common: &OperationCommon, segments: &[ToolpathSegment]) -> Vec<NCBlock>;
    fn validate(&self, common: &OperationCommon) -> Result<(), Error>;
}

pub enum OperationKind {
    Drill(Drill),
}

pub struct Operation {
    pub common: OperationCommon,
    pub kind: OperationKind,
}

impl Operation {
    fn kind_impl(&self) -> &dyn OperationType {
        match &self.kind {
            OperationKind::Drill(d) => d,
        }
    }

    pub fn generate(&self) -> Vec<ToolpathSegment> {
        self.kind_impl().generate(&self.common)
    }

    pub fn compile(&self, segments: &[ToolpathSegment]) -> Vec<NCBlock> {
        self.kind_impl().compile(&self.common, segments)
    }

    pub fn validate(&self) -> Result<(), Error> {
        self.kind_impl().validate(&self.common)
    }
}
```

Move `DrillStrategy`, `OperationLocations`, and `Pattern` here from their current locations.
