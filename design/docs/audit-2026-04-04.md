---

## High Priority

### `io/job.rs:32-35` ŌĆö `process::exit()` inside library code

```rust
let pp_lua = std::fs::read_to_string(&pp_path).unwrap_or_else(|e| {
    eprintln!("Error reading post-processor: {}", e);
    std::process::exit(1);  // library must never do this
});
```

Library code must propagate errors, not call `exit()`. This makes `run_job()` untestable and prevents any future caller (REST API, test harness) from handling the error. Return `Err(anyhow!(...))` instead.

### `io/job.rs:16` ŌĆö `unwrap()` on `file_stem()` can panic

```rust
Path::new(path).file_stem().unwrap().to_string_lossy().into_owned()
```

Panics on paths like `.` or paths with no stem. Use:

```rust
Path::new(path)
    .file_stem()
    .and_then(|s| s.to_str())
    .unwrap_or("unnamed")
    .to_string()
```

### `core/pattern.rs:68-69` ŌĆö integer overflow on `i32::MIN` count

```rust
(0..n.abs())  // i32::MIN.abs() overflows ŌåÆ panics in debug, wraps in release
```

The `n == 0` guard is present, but `i32::MIN.abs()` still overflows. Add:

```rust
if *n == 0 || *n == i32::MIN {
    return Err(anyhow!("Circular pattern: count must be a non-zero i32"));
}
```

## Medium Priority

### `io/job.rs:61` ŌĆö unnecessary clone of `Option<String>`

```rust
job.name.clone().unwrap_or("unnamed".to_string())
```

Clones the entire `Option<String>` before inspecting it. Prefer:

```rust
job.name.as_deref().unwrap_or("unnamed").to_string()
```

### `nc/bridge.rs:12, 40` ŌĆö `BASE_LUA` allocated to `String` twice per job

`BASE_LUA` is `&'static str`. Both `get_capabilities()` and `generate_nc()` call `.to_string()` on it, each allocating a fresh `String`. Consider a shared helper or `once_cell::sync::Lazy<String>` if mlua truly requires an owned value.

### `core/pattern.rs:72` ŌĆö direct `== 0.0` float comparison

```rust
if *step_deg == 0.0 { ... }
```

Inconsistent with the epsilon comparisons in the same file. Use `step_deg.abs() < 1e-9`.

### `nc/ir.rs:8-9` ŌĆö `NCState` fields are private on a public struct

```rust
pub struct NCState {
    spindle: SpindleState,  // private
    coolant: bool,          // private
}
```

The struct is public and serialized to Lua, but callers cannot inspect its fields. Make them `pub` or add accessor methods.

### `nc/ir.rs` ŌĆö `NCState` should derive `Copy`

Both fields (`SpindleState` enum, `bool`) are `Copy`. Deriving `Copy` eliminates the `state.clone()` call in `ir.rs:135` and makes the intent clearer.

### `io/parsing.rs` ŌĆö YAML types unnecessarily public

`JobConfig`, `OperationConfig`, and `CommonOperationConfig` are YAML-specific parsing internals. They are currently part of the public API but should not be. Mark `parsing.rs` as `pub(crate)` and expose only `load_job()` and `run_job()`.

### Core types missing `PartialEq`

`NCBlock`, `ToolpathSegment`, `Pattern`, and `Tool` all lack `PartialEq`. This makes it impossible to write `assert_eq!` in tests. Should be added sooner rather than later as test coverage grows.

### Tests: two fixture files have no expected output

`tests/nc_output.rs` references `drill_circular.yaml` and `drill_multiple.yaml` but there are no corresponding `.h` output files. These tests silently do nothing useful until expected outputs are added.

### `core/pattern.rs` ŌĆö radius resolution duplicated between `into_points()` and `into_segment()`

The `(diameter, radius)` match block (lines 39ŌĆō53 and 120ŌĆō133) is copy-pasted verbatim in both methods. If the logic or error message changes in one, the other silently diverges. Extract to a private helper:

```rust
fn resolve_radius(diameter: Option<f64>, radius: Option<f64>) -> Result<f64> { ... }
```

## Low Priority / Cleanup

**`Cargo.toml` ŌĆö `thiserror` declared but never imported.** Remove it, or start using it for typed errors at public API boundaries.

**`cli/mod.rs` is empty.** Per the stated design ("cli/ and api/ are peer-level thin adapters"), CLI logic should live in `src/cli/` and be called from `bin/chipmunk.rs`. Low urgency until the API binary is added.

**Quill and Drill are identical structs** with nearly identical `generate()` logic:

```rust
pub struct Drill { pub locations: Locations }
pub struct Quill { pub locations: Locations }
```

Only `compile()` differs. Share the `generate()` implementation via a free function to avoid future divergence.

**`CLAUDE.md` dependency rule is inaccurate.** It states `io/ ŌåÆ core/`, but `io/` also imports `nc/` and `operations/`. The code is correct; the documentation is wrong. The rule should read `io/ ŌåÆ core/, nc/, operations/`.

**`String::from("...")` vs `.to_string()`** ŌĆö `quill.rs:47` uses `String::from` while the rest of the codebase uses `.to_string()`. Minor consistency issue.

**`ToolpathSegment` carries an embedded `NCBlock`** via `pattern: Option<NCBlock>`. This mixes two abstraction layers ŌĆö a toolpath segment that can contain an IR block. The pattern is intentional (passing a `PatternCircular` block through to `compile()` without expanding it), but a future reader won't understand why without a comment explaining the design decision.

## Future Problems Worth Addressing Before They Become Painful

### Arc toolpath fields on `ToolpathSegment`

`NCBlock::ArcCw` now has `i`, `j`, and `r` fields ŌĆö good. `ToolpathSegment` itself has no arc fields; arc moves would currently need to be threaded through the `pattern: Option<NCBlock>` field. Worth revisiting when arc toolpaths are generated in Phase 4 ŌĆö at that point it may be cleaner to add `i`, `j`, `r` directly to `ToolpathSegment` or introduce a dedicated arc variant.

### Lua VM lifecycle

`get_capabilities()` and `generate_nc()` each create a fresh `Lua::new()` and re-load `base.lua`. Fine for one-off CLI use. If batching operations is ever added, a struct wrapping a single configured Lua VM would be the right abstraction and would also eliminate the `BASE_LUA` double-allocation issue.

### Parallelization TODO

`job.rs:47` has a comment suggesting `rayon::par_iter` on the operation loop. The Lua bridge is irrelevant here ŌĆö that loop only calls `op.generate()` and `op.compile()`, which are pure Rust (geometry math and `NCBlock` construction). Lua is only invoked once after all operations are compiled, in `generate_nc()` at line 58.

The actual constraint is that `capabilities` (a `&PostprocessorCapabilities` shared across all iterations) must be `Sync`. That is likely already true or trivially achievable. `rayon::par_iter` should be usable here without any Lua-related concerns.

### No input validation after YAML parsing

The "never infer, always error" principle is not enforced by the schema. These values are accepted silently and only fail later (or produce nonsensical NC output):

- `clearance: f64` ŌĆö can be negative
- `spindle_speed: f64` ŌĆö can be zero or negative
- `diameter: f64` ŌĆö can be zero (degenerate pattern)
- `points: Vec<...>` ŌĆö can be empty

Validation at parse time produces clearer error messages than silent failure downstream.

## What Is Already Good

- No `unsafe` blocks anywhere
- No `panic!()`, `todo!()`, or `unimplemented!()` in production paths
- Correct lifetime annotations on `Operation<'a>` and `AnnotatedBlock<'a>`
- Consistent `anyhow::Result` + `?` error propagation throughout
- Modern Option/Result combinators used correctly (`is_some_and`, `ok_or_else`, `and_then`)
- Epsilon-guarded float comparisons in loop termination logic (`pattern.rs:78-80`)
- No circular dependencies ŌĆö module graph is a clean DAG
- `Pattern` correctly split into its own `core/pattern.rs` file
- `NCBlock::ArcCw` has full arc context (`i`, `j`, `r`) so post-processors can pick what they need
